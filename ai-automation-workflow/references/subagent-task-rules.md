# AI自动化工作流子Agent任务规范

本文件是 `ai-automation-workflow` 内部的子 Agent 必读规范页。它不是所有工作流的全局规范，除非用户明确要求复用。

## 核心规则

子 Agent 不共享主 Agent 的上下文。本工作流中每个派发任务都必须自包含。

提示词必须使用下面两种输入方式之一：

- **详单模式**：直接给行号、标题、问题描述、截图、当前方案/批注、项目目录和约束。
- **文档索引模式**：给本地文档路径、准确的 row/section、截图目录或图片路径、项目目录和约束。

不要只给 row 编号而不给文档路径或问题内容。

## 必带规范块

每次派发子 Agent 时，都把下面这段贴进任务包：

```text
通用规范（每次任务都必须遵守）：
- 你和主 agent 不共享上下文；只能依据本提示、给定文档、截图和目标项目自行查证。
- 只处理明确分配给你的 row/section，不要顺手分析或修改其他条目。
- 不要直接修改主任务 Markdown、方案 Markdown、`rows.json`、payload 或腾讯文档；你只输出结构化调查结果，主 Agent 复核后统一写入。
- 如果本任务包包含太多跨系统条目、缺少具体问题内容、缺少文档路径或截图，请返回 `状态：信息不足` 并说明任务包需要拆分或补充什么，不要自行扩大/重分组。
- 如果任务包是大范围行号、批量文本分类或要求你一次处理几十条，请拒绝该任务包，要求拆成小批次；不要硬做。
- 如果任务要求你输出 TSV 替换行或批量覆盖主文档，请拒绝该格式，改为输出逐条结构化分析块。
- 先读工作文档中对应条目和截图；有截图必须结合截图判断。
- 代码问题要查真实类/方法/调用链，不能只按标题猜。
- Prefab/Scene/UI 问题可以读 YAML、Prefab 路径和组件引用；不要建议手改 raw YAML，方案写 Unity 里需要调整的对象、组件、绑定、层级或 RectTransform。
- 资源问题先判断是否是逻辑引用、显示状态、层级、材质、动画或特效切换错误；只有证据指向资源本身时才建议补图/换图/改材质/动画/特效。
- 配置问题要写具体配置表、字段、行或 ID；不要只写“检查配置”。
- 不要泛化总结，不要套无关规范，不要把无关问题归到存档、LFActionRunner 或其他不相关系统。
- `状态：信息不足` 主要用于以下情况：优化调整项的意图不明确；不知道这条到底想改成什么；不知道期望表现是什么；有多个合理修法但拿不定该选哪一个。不要瞎想，不要硬凑方案。
- 如果按需求描述应该存在某个入口、文件、Prefab、资源、配置或模块，但在目标项目中找不到，也输出 `状态：信息不足`。这表示需求和项目现状对不上，无法判断怎么改。
- 这类 `信息不足` 必须写清：查了哪些路径/关键词；没找到什么；需要用户确认真实入口、资源名、配置 ID 或模块名。
- `信息不足` 时必须说明还缺哪类产品/表现信息：期望表现、目标交互、前后对比、取舍规则、具体场景、参考图/视频、配置 ID 或对象名。只写“需要实机确认”不够，必须说明确认什么。
- 如果问题意图清楚，且已经有具体方案，只是缺少 Unity 运行态、Prefab 实际绑定、日志或复现证据，应输出候选入口、具体方案和 `方案待确认`。如果没有具体方案，不要把“缺证据”写成 `方案待确认`。
- 只有已经有具体修法时，才写验证方式；信息不足时不要伪造验收/验证步骤。
```

## 必须提供的输入

每个任务必须说明：

- 工作模式：只读分析、用户批注修订或实施修复。
- 工作目录或目标项目路径。
- 分配的 row/section。
- 来源文档路径，或完整问题文本。
- 截图路径或 URL。
- 修订/实施时的当前方案和用户批注。
- 实施任务的允许修改边界。

## 必须要求的输出

分析/修订任务必须输出：

- 真实入口：代码类/方法、Prefab/Scene YAML、资源、配置/CSV 行。
- 调用链或逻辑判断。
- 根因。
- 具体修改方案。
- 涉及代码 / Prefab / 资源 / 配置。
- 缺少的信息。

如果主 Agent 要把结果回填腾讯文档，子 Agent 只提供素材，不直接改写主文档或 payload。建议按下面三块提供可复核内容，方便主 Agent 逐条整理：

- `原因分析素材`：Bug 类写根因证据；设计向、优化向、体验调整类写现状、调整意图和目标，不要硬编 Bug 根因。
- `修改方案素材`：按本条实际涉及的代码、Prefab/UI、资源、配置写具体改法。
- `待用户确认/信息不足素材`：有方案且确实存在会影响最终取舍的用户决策点时写具体确认问题；无方案时写信息不足原因和需要用户补充什么。单纯“等用户确认方案”不要写成素材。

实施任务必须输出：

- 修改了哪些文件。
- 每个文件为什么改。
- 做了什么验证。
- 剩余风险或阻塞项。

## 主 Agent 复核

主 Agent 必须复核子 Agent 输出后再使用：

- 核对引用的文件、类、配置行是否存在。
- 拒绝“检查配置”“看一下 Prefab”这类空泛输出。
- 拒绝无关重构或超出分配 row 的工作。
- 把不确定项转成明确缺失证据。
- 不要把未经复核的子 Agent 文本直接贴进最终文档。
