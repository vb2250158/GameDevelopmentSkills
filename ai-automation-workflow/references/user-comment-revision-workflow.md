# 用户批注修订工作流

第一轮方案回填后，用户会在 `修正方案批注` 写审核意见。本流程用于把这些批注吸收到本地方案并重新分析。

## 批注含义

`修正方案批注` 是用户字段，可能包含纠错、缺失上下文、不同意见、额外证据或要求重新检查某一行。

不要把这个字段当作 Agent 状态字段。正常修订流程中不要覆盖或清空它。

## 步骤

1. 重新读取腾讯文档表头和当前目标行。
2. 把非空 `修正方案批注` 拉取到本地 review Markdown。
3. 对每条批注对比：
   - 当前线上 `修正方案`
   - 当前 `确认修正方案`
   - 用户批注
   - 本地方案文档小节
   - 截图和原始任务行
4. 判断主 Agent 自己能否解决，还是需要子 Agent。
5. 必要时重新查项目证据：代码、调用链、配置、Prefab/Scene YAML、资源引用、截图和相关文档。
6. 更新本地方案文档中的根因、修改方案；只有确实存在会影响最终取舍的用户决策点时，才补充待用户确认项。
7. 重新生成 payload，只回填 `修正方案` / `确认修正方案` / 用户明确要求的分配字段。

使用子 Agent 时，要么粘贴完整问题包，要么给批注文档路径和准确 row/section。不要假设子 Agent 能看到主线程或记得原始腾讯文档。

## 子Agent补充提示

某行有用户批注时，在普通子 Agent 模板后追加下面内容。每次提示词还必须要求子 Agent 阅读 `C:\Users\%USERNAME%\.codex\skills\ai-automation-workflow\references\subagent-task-rules.md`，并重复 `subagent-task-template.md` 中的必带规范块。

```text
用户批注：
<用户批注>

当前修正方案：
<当前修正方案>

可选文档索引：
- 批注文档：<批注复核文档路径>
- 原始清单/方案文档：<任务或方案文档路径>
- 负责条目：<行号列表>

请重点判断：
1. 用户批注是否指出当前方案错误、过泛、缺入口、缺资源/Prefab/配置细节，还是只是补充信息。
2. 基于批注重新查证代码/配置/Prefab/资源后，当前方案应该如何改。
3. 如果仍信息不足，明确还缺什么证据；不要把“待用户确认”误写成已能修改。
```

## 修订后的确认状态

- `方案待确认`：修订后的方案具体，可交给用户审核确认。
- `信息不足`：结合批注并重新查证后，仍无法产出具体修法。

## 停止条件

遇到下面情况，先停止，不要 live backfill：

- 无法安全映射到当前线上行。
- 用户批注和本地截图/方案互相矛盾，且没有可靠来源能解决。
- 同一行多个批注暗示不同产品意图。
- 所需证据必须 Unity Play Mode 或人工复现，而用户尚未授权。
