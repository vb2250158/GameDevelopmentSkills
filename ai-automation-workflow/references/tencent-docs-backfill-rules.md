# 腾讯文档回填规则

## 列规则

每次回填前都先读取第 1 行表头。按表头名映射列，不要硬编码列字母。

尽量按稳定的行内容映射行。优先用本地 Markdown 标题和当前线上表格的 `描述` 做精确匹配。表格排序、筛选、插行或编辑后，不要假设 Markdown 行号仍然等于当前物理行号。

必需表头：

- `修正方案`：本地 Markdown 文档中的已复核方案文本。
- `确认修正方案`：Agent 写入的方案状态。
- `修正方案批注`：用户拥有的批注字段；除非用户明确要求，否则保持为空或保留原内容。

可选表头：

- `负责人`：只有用户要求分配任务时才写。
- `状态`、`优先级`、`描述`：用于筛选和校验。

## 状态筛选

回填前必须读取 `状态` 列并确认目标范围。

- 用户说“没有修正方案 / 未写修正方案 / 待出方案 / 待处理”但没有指定状态时，默认只处理 `A未开始`。
- `D已完成` 默认不需要出修正方案，也不要写 `方案待确认` 或 `信息不足`。
- `B进行中`、`C待验收` 只有用户明确要求包含时才处理。
- 如果本地任务包或 payload 里混入 `D已完成`，先停止并剔除这些行；只有用户明确要求“补历史方案/复盘已完成项/已完成也回填方案”时才允许写入，并在报告里标明这是例外。
- `修正方案` 是否为空是 AI 是否已经接入该条优化调整项的主标记：为空表示未接入/待出方案；非空表示已接入过，需要走审核、批注修订或确认流程。
- 回填新方案时，默认目标是 `状态 = A未开始` 且 `修正方案` 原本为空的行。修订已有方案时，默认目标是 `状态 = A未开始` 且 `修正方案` 非空的行。

## 状态值

`确认修正方案` 只能按下面口径写：

- `方案待确认`：Agent 已有具体可执行修法，只等用户审核确认。
- `方案待确认` 只能用于有解方案的条目：已经说明要改什么代码、Prefab/UI、资源或配置，并且可交给工程师执行。
- `信息不足`：没有具体解决方案，或只有“不确定/需要用户确认/需要复现/需要看日志/需要确认”的空壳说明，或方案空泛到不能交给工程师修改。
- `已确认方案`：只能由用户审核后确认，Agent 不要自行把 `方案待确认` 改成 `已确认方案`。
- `处理中`：实施阶段主 Agent 已经领取并分配处理。主 Agent 写入该值后才能启动实施子 Agent；子 Agent 不直接写腾讯文档。
- `已处理完成`：实施完成后，主 Agent Review Check 通过并读回校验后写入；写入时必须同步把表格 `状态` 改成 `C待验收` 并读回校验。不要在子 Agent 刚返回时直接写完成。
- `关闭`：用户关闭该项；后续拉取、分析、实施和回填都默认忽略。
- 如果需求描述指向的文件、Prefab、资源、配置、模块或入口在目标项目中找不到，也写 `信息不足`，并在方案文本里写清查了哪些路径/关键词、没找到什么、需要用户确认什么。
- 单条状态由负责分析该条的子 Agent 先给建议；主 Agent 复核后再回填。主 Agent 不要在缺少子 Agent 证据时批量给状态。

不要在这个工作流中写 `否`，除非用户再次明确改变表格口径。

实施阶段状态流转固定为：

```text
已确认方案 -> 处理中 -> 已处理完成，并同步 状态 -> C待验收
```

`关闭` 是终态忽略；`处理中`、`已处理完成` 都不能再次作为待实施任务导出。

## `修正方案` 单元格格式

`修正方案` 由主 Agent 逐条整理后写入。可以批量写入腾讯文档，但写入内容必须逐条由主 Agent 确认。脚本只负责搬运、定位行列和校验，不负责自动生成原因分析、修改方案或状态说明；不要把子 Agent 原文、TSV、日志片段或一串“待确认：...”直接贴进单元格。

每条可以使用下面这种结构，但段落按任务类型取舍。Bug、异常、缺失、错误表现、回归、配置失效类通常写 `【原因分析】`；策划、设计、内容、体验、文案/布局/数值偏好类不一定需要原因段，不要为了格式硬编根因。

```text
【原因分析】
这条静态未发现已修支持。当前 <代码/Prefab/资源/配置链路> 仍然 <现状>，所以 <为什么会导致问题>。
设计/优化类如需原因段，可写：这条属于 <设计/优化/体验/内容> 调整。当前 <截图/配置/现有表现> 是 <现状>，期望调整为 <目标方向>。

【修改方案】
- 代码：修改 <类/方法>，<具体逻辑调整>。
- Prefab/UI：调整 <窗口/节点/组件/绑定/层级/RectTransform>。
- 资源：<需要补图/换图/材质/动画/特效修正；若未确认资源本身问题，写清资源项仍需用户确认的具体原因>。
- 配置：修正 <CSV/字段/ID/跳转目标/文案>。

【待用户确认】
只有确实存在会影响最终取舍的用户决策点时才写，例如两种方案需要选一种、缺少资源名/配置 ID、需要产品确认具体数值或表现标准。单纯“等待用户确认方案”“需要验收看看效果”“Unity 运行后复核”不要写成正文段落。
```

不是每条都要写满代码、Prefab/UI、资源、配置四类；只写本条实际涉及的项。`【修改方案】` 必须能交给工程师执行，不能只有“检查一下”。
有具体可执行方案时，`修正方案` 正文通常只需要 `【原因分析】` 和 `【修改方案】`，`确认修正方案` 状态列单独写 `方案待确认`。不要为了匹配状态而在正文末尾硬加 `【待确认】`、`【方案待确认】` 或类似段落。

如果没有具体解决方案，最后一段改为 `【信息不足】`，并且 `【修改方案】` 不要硬凑。信息不足时第一段仍写 `【原因分析】`：

```text
【原因分析】
信息不足：需求描述指向 <某文件/Prefab/资源/配置/模块/表现>，但静态查找 <路径/关键词> 未找到对应入口，当前无法判断真实改动点。

【修改方案】
暂无可执行方案。

【信息不足】
需要用户补充 <期望表现/真实入口/资源名/配置 ID/模块名/参考图或视频/取舍规则>。
```

`确认修正方案` 状态列按下面口径写：

- 有具体可执行方案，只等用户审核确认：`确认修正方案` 写 `方案待确认`，正文不必追加状态段。
- 有具体可执行方案，但确实存在会影响最终取舍的用户决策点：正文可追加 `【待用户确认】` 并写具体问题，`确认修正方案` 仍写 `方案待确认`。
- 不知道要改什么、期望表现不清、多个合理方案无法取舍，或需求指向的入口/资源/配置不存在：正文写 `【信息不足】`，`确认修正方案` 写 `信息不足`。

## 替换安全

腾讯文档写入可能因为单元格状态表现为追加。需要替换时：

1. 读取目标区域。
2. 只清空需要替换的目标行和目标列。
3. 写入新值。
4. 读回并和预期值逐格比较。

目标行稀疏时，不要清整列。要按连续行段构造清空范围。

如果 payload 生成出现行号回退警告，先停止，不要 live write；需要人工或主 Agent 先确认这些行映射。写入后如果校验出现异常行，也不要继续写下一批，先搞清不匹配原因。

## 修正方案变更留痕

`修正方案批注` 默认是用户字段，不写 Agent 状态。但当 Agent 修改了线上已经存在的 `修正方案` 时，必须在对应行 `修正方案批注` 末尾追加变更说明，方便用户知道方案被改过。

规则：

- 首次给空白 `修正方案` 写入方案时，不需要写变更留痕。
- 覆盖、重写、补充或调整已有 `修正方案` 时，必须保留原批注，并在末尾追加：

推荐格式：

```text
[Agent修订：<简短说明本次改动，例如“补充 Prefab 入口和待用户确认点”“把状态从方案待确认改为信息不足，因为缺少真实资源名”>]
```

也可以使用旧格式：

```text
[写入改了什么] <简短说明本次改动>
```

- 如果原批注非空，先追加一个空行，再写 `[Agent修订：...]`。
- 后续看到 `修正方案批注` 里有 `[xxx]` 形式的方括号标记，默认表示该条方案已经被 Agent 修订过。再次读取批注、修订方案或回填时，要把这些标记当作变更历史阅读，不要忽略。
- 不要把 `方案待确认`、`信息不足`、`处理中` 这类状态单独写进 `修正方案批注`；状态写在 `确认修正方案`。

示例：

```text
这个是解锁新动物 unlocknew xxx，好像是那个 tween 被错跑了

[Agent修订：将方案从运动会通用点击提示修订为 UnlockNewNPCWindow / Breath 组 tween 已在 Prefab 中但未由控制器播放，需要绑定/播放 breathTween]
```

## 校验报告

写入后报告：

- 检查了多少目标行。
- `修正方案` 精确匹配数量。
- `方案待确认` 和 `信息不足` 数量。
- `修正方案批注` 是否在目标行保持空白或未被 Agent 覆盖。
- 异常行列表：行号和不匹配列。
