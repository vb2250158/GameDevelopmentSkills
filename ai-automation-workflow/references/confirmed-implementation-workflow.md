# 已确认方案实施工作流

用户确认具体方案后，才进入实施阶段。

## `确认修正方案` 状态机

这一列同时承担方案审核和实施流转状态：

- `方案待确认`：Agent 已给出具体可执行方案，等待用户审核确认；不能实施。
- `信息不足`：当前方案不足以实施；不能实施。
- `已确认方案`：用户已确认，可以由主 Agent 领取并分配实施。
- `处理中`：主 Agent 已经把该行领取并分配给自己或子 Agent；不要重复领取。
- `已处理完成`：主 Agent Review Check 通过，腾讯文档确认列已完成；写入该值时必须同步把表格 `状态` 改成 `C待验收`，表示等待人工验收；后续不再作为 Agent 待实施项关注。
- `关闭`：用户关闭该项；后续不再关注。

主 Agent 负责状态流转。子 Agent 不直接改腾讯文档。

## 进入条件

只有同时满足下面条件，某一行才能实施：

- `确认修正方案` 精确等于 `已确认方案`。
- `修正方案` 包含具体实施计划。
- 该行能安全映射到当前线上表格和本地方案小节。

`方案待确认` 表示还在等用户审核；`信息不足` 表示还需要补充分析。除非用户明确覆盖规则，否则这两种状态都不能实施。
`处理中` 表示已经被领取，不要重复派发；`已处理完成` 和 `关闭` 都是终态，默认忽略。

## 步骤

1. 重新读取腾讯文档表头，导出已确认行。
2. 主 Agent 按具体任务拆分实施包；只有修复触及同一个模块、Prefab 或配置时，才按系统分组，否则一行一个实施任务。
3. 启动子 Agent 前，主 Agent 必须先把这些目标行的 `确认修正方案` 从 `已确认方案` 改成在线表下拉项 `处理中`，并读回确认。写入失败时不要启动子 Agent，避免重复领取。
4. 仅对互相独立的行或独立分组并行启动子 Agent。
5. 给每个子 Agent 严格的实施任务包：
   - row id 和当前线上标题
   - 已确认的 `修正方案`
   - 用户 `修正方案批注`
   - 截图或本地图片路径
   - 目标项目目录
   - 方案里已识别的文件/模块
   - 允许修改边界和必须验证的内容
   - 或者给已确认方案文档路径，以及准确分配的 row/section
6. 主 Agent 必须 review 所有子 Agent diff。拒绝无关重构、过大改写、未经允许的 raw YAML 大改，以及超出分配 row 的改动。
7. 按触及区域跑验证：静态编译、定向测试、Unity MCP 检查、配置解析检查、截图或人工验证等。
8. Review Check 没问题后，主 Agent 把对应行的 `确认修正方案` 从 `处理中` 改成 `已处理完成`，同时把表格 `状态` 从 `A未开始` 改成 `C待验收`，并读回校验这两列。
9. 汇总已修行、修改文件、验证结果和未解决行。未通过复核的行不要标完成；写清阻塞并保留 `处理中` 或按用户指示调整。

## 实施子Agent提示模板

每次都要求子 Agent 阅读 `C:\Users\%USERNAME%\.codex\skills\ai-automation-workflow\SKILL.md` 和 `C:\Users\%USERNAME%\.codex\skills\ai-automation-workflow\references\subagent-task-rules.md`。同时包含 `subagent-task-template.md` 中的必带规范块，再追加下面实施规则。

```text
先阅读并遵守这些规范文档：
C:\Users\%USERNAME%\.codex\skills\ai-automation-workflow\SKILL.md
C:\Users\%USERNAME%\.codex\skills\ai-automation-workflow\references\subagent-task-rules.md

实施修复 row <行号>，目标项目 <项目目录>。

只处理这一条/这一组已确认方案，不要扩大范围。

输入：
- 标题：<标题>
- 用户确认状态：<确认修正方案>
- 修正方案：<已确认修正方案>
- 用户批注：<用户批注或空>
- 截图：<本地截图路径或 URL>
- 允许修改范围：<允许修改的文件或模块>

或文档索引输入：
- 已确认方案文档：<已确认方案文档路径>
- 负责条目：<行号列表或小节列表>
- 目标项目：<项目目录>
- 允许修改范围：<允许修改的文件或模块>

要求：
1. 先复核方案涉及的代码/Prefab/资源/配置入口是否真实存在。
2. 按已确认方案实施最小必要修改。
3. 不要改无关系统，不要做顺手重构。
4. 涉及 Prefab/Scene 时优先用项目 Unity 工具或安全流程；不要直接大范围手改 YAML。
5. 完成后报告：改了哪些文件、每个文件为什么改、如何验证、还有什么风险。
6. 如果实施中发现方案不成立，停止扩大修改范围，报告缺少的信息或需要主 Agent 决策的点。
```

## 主Agent复核清单

- 每个改动文件都对应已确认行。
- 实现遵循已确认方案；如有偏离，必须有明确理由。
- 没有误实施 `方案待确认` / `信息不足` 行。
- 没有误实施 `处理中`、`已处理完成` 或 `关闭` 行。
- 没有覆盖用户拥有的腾讯文档批注。
- 验证输出是当前的，并且和改动文件相关。
- 只有主 Agent Review Check 通过后，才把 `确认修正方案` 改成 `已处理完成`，并同步把表格 `状态` 改成 `C待验收`；未通过复核时两列都不要写完成态。
