# 子Agent任务模板

派发聚焦分析任务时使用这个模板。子 Agent 不共享主 Agent 上下文，所以提示词必须自包含。

允许两种输入包：

- **详单模式**：把标题、问题描述、截图、当前方案/批注和项目目录直接写进提示词。
- **文档索引模式**：提供本地文档路径，以及子 Agent 负责的准确 row 或 section。文档中必须包含标题、截图和当前方案。

不要只给 row 编号而不给文档路径或问题内容。

## 派发粒度

- 优先一次 1 条任务。
- 只有同一窗口、同一配置表、同一 Prefab、同一代码入口或同一复现链路的强相关问题，才可以合并 2-5 条。
- 禁止把几十条跨系统问题按主 Agent 猜测的大类合成一个任务包，例如“登录/性能/建造/钓鱼/剧情/UI 混合包”。
- 禁止给子 Agent `2-120`、`121-240` 这种大范围文本复核任务；这不是逐条方案分析。
- 如果总量很大，主 Agent 必须切成 5-10 条的小批次，并为每条提供标题、截图、当前方案/批注和输出位置。
- 一批内可以并行多个子 Agent；推荐每个子 Agent 负责 1 条，或负责 2-5 条强相关小组。
- 禁止要求子 Agent 输出 TSV 替换行。子 Agent 应输出逐条结构化分析块，主 Agent 复核后再负责写入 Markdown/JSON/payload。
- 子 Agent 不直接改主任务 Markdown、方案 Markdown、`rows.json`、payload 或腾讯文档；最终写入由主 Agent 复核后完成。
- 主 Agent 不要先替子 Agent 做大类归纳并承诺“先出第一版方案框架”。子 Agent 返回前，只能说明调度范围和等待证据。

每个子 Agent 提示词开头都要要求它先阅读本 Skill 内的规范文档：

```text
先阅读并遵守这些规范文档：
C:\Users\%USERNAME%\.codex\skills\ai-automation-workflow\SKILL.md
C:\Users\%USERNAME%\.codex\skills\ai-automation-workflow\references\subagent-task-rules.md
```

同时在任务包中重复下面的规范块，作为兜底。不要假设子 Agent 已经加载主 Agent 的 Skill 上下文。

## 必带规范块

```text
通用规范（每次任务都必须遵守）：
- 你和主 agent 不共享上下文；只能依据本提示、给定文档、截图和目标项目自行查证。
- 只处理明确分配给你的 row/section，不要顺手分析或修改其他条目。
- 如果本任务包包含太多跨系统条目、缺少具体问题内容、缺少文档路径或截图，请返回 `状态：信息不足` 并说明任务包需要拆分或补充什么，不要自行扩大/重分组。
- 如果任务包是大范围行号、批量文本分类或要求你一次处理几十条，请拒绝该任务包，要求拆成小批次；不要硬做。
- 如果任务要求你输出 TSV 替换行或批量覆盖主文档，请拒绝该格式，改为输出逐条结构化分析块。
- 先读工作文档中对应条目和截图；有截图必须结合截图判断。
- 代码问题要查真实类/方法/调用链，不能只按标题猜。
- Prefab/Scene/UI 问题可以读 YAML、Prefab 路径和组件引用；不要建议手改 raw YAML，方案写 Unity 里需要调整的对象、组件、绑定、层级或 RectTransform。
- 资源问题先判断是否是逻辑引用、显示状态、层级、材质、动画或特效切换错误；只有证据指向资源本身时才建议补图/换图/改材质/动画/特效。
- 配置问题要写具体配置表、字段、行或 ID；不要只写“检查配置”。
- 不要泛化总结，不要套无关规范，不要把无关问题归到存档、LFActionRunner 或其他不相关系统。
- `状态：信息不足` 主要用于以下情况：优化调整项的意图不明确；不知道这条到底想改成什么；不知道期望表现是什么；有多个合理修法但拿不定该选哪一个。不要瞎想，不要硬凑方案。
- 如果按需求描述应该存在某个入口、文件、Prefab、资源、配置或模块，但在目标项目中找不到，也输出 `状态：信息不足`。这表示需求和项目现状对不上，无法判断怎么改。
- 这类 `信息不足` 必须写清：查了哪些路径/关键词；没找到什么；需要用户确认真实入口、资源名、配置 ID 或模块名。
- `信息不足` 时必须说明还缺哪类产品/表现信息：期望表现、目标交互、前后对比、取舍规则、具体场景、参考图/视频、配置 ID 或对象名。只写“需要实机确认”不够，必须说明确认什么。
- 如果问题意图清楚，且已经有具体方案，只是缺少 Unity 运行态、Prefab 实际绑定、日志或复现证据，应输出候选入口、具体方案和 `方案待确认`。如果没有具体方案，不要把“缺证据”写成 `方案待确认`。
- 只有已经有具体修法时，才写验证方式；信息不足时不要伪造验收/验证步骤。
```

## 只读分析模板

```text
先阅读并遵守这些规范文档：
C:\Users\%USERNAME%\.codex\skills\ai-automation-workflow\SKILL.md
C:\Users\%USERNAME%\.codex\skills\ai-automation-workflow\references\subagent-task-rules.md

只读分析 row <行号>，目标项目 <项目目录>，不要修改任何文件，不启动 Unity。

输入（详单模式）：
- 行号：<行号>
- 标题：<标题>
- 现象/内容说明：<问题描述>
- 截图：<本地截图路径或 URL>
- 相关本地清单：<任务清单 Markdown 路径>
- 只读约束：不要修改任何文件，不启动 Unity，除非主 agent 明确授权。
- 分析范围：代码 / Prefab YAML / 资源 / 配置（按本条需要列明）。

或输入（文档索引模式）：
- 工作文档：<任务或方案文档路径>
- 负责条目：row <行号列表> / section <小节列表>
- 截图目录：<截图目录或图片列表>
- 目标项目：<项目目录>

要求逐条回答：
0. 状态：`可给方案`、`方案待确认` 或 `信息不足`。`信息不足` 用于需求/优化意图不清、方案取舍无法判断，或需求描述指向的文件/入口/资源/配置在项目中找不到导致现状对不上；`方案待确认` 只能用于已有具体可执行方案、等待用户审核确认的条目，不表示 Agent 还没想清楚方案。如果是 `信息不足`，只回答缺失的产品/表现信息、找不到的项目入口和已尝试理解的方向，不要继续编造根因和方案。
1. 真实入口：具体代码类/方法、Prefab/Scene YAML、资源、配置/CSV 候选入口；写出为什么是这些入口。
2. 调用链和根因：逻辑问题要分析具体代码路径和调用栈；Prefab/资源问题要说明 YAML/引用/绑定证据；配置问题要说明字段和行。
3. 具体修改方案：需要修改什么代码、什么 Prefab/UI、什么资源、什么配置；写到可交给工程师修改的粒度。
4. 涉及范围：明确本条涉及代码、Prefab、资源、配置中的哪些项。
5. 无法确定项：如果只是局部不确定，说明缺什么证据，需要主 agent 或用户打开哪个窗口、复现哪一步、看哪个 Prefab 绑定或哪张配置表。
6. 验证方式：仅在 `状态：可给方案` 时提供。`状态：信息不足` 时不要写验证方式。

如果本条会用于腾讯文档回填，请额外提供三块素材，供主 Agent 复核后整理成单元格文本：

```text
【原因分析素材】
<Bug 类写根因证据；设计向、优化向、体验调整类写现状、调整意图和目标，不要硬编 Bug 根因>

【修改方案素材】
- 代码：<如涉及>
- Prefab/UI：<如涉及>
- 资源：<如涉及>
- 配置：<如涉及>

【待用户确认素材】或【信息不足素材】
<有方案且确实存在会影响最终取舍的用户决策点时，写具体需要用户确认的问题；无方案则写缺少什么信息。单纯“等待用户确认方案”不用写素材。>
```

不要把这三块当成最终回填文本；最终写入 `修正方案` 的内容由主 Agent 复核后逐条整理。
```

## 主Agent复核清单

- 核对子 Agent 提到的文件、类、配置行是否存在。
- 拒绝“检查配置”这类没有表名、字段、行号的方案。
- 把“疑似”转成具体结论；如果连问题含义、入口或修法都不清楚，转成 `信息不足`，并写清需要用户补充什么。
- 只有已经有具体可执行修正方案时，最终回填状态才能是 `方案待确认`；没有解决方案的一律归 `信息不足`，不要把“不确定”写成 `方案待确认`。
- 单条状态初判由子 Agent 给出；主 Agent 只做复核和纠偏。不要在子 Agent 调查前替所有条目集中判状态。
- 主 Agent 不要在用户要求子 Agent 分析时抢先完成子 Agent 的调查；只做任务拆分、必要核对、复核和汇总。
- 不要把未经复核的子 Agent 输出直接写进最终文档。
