# 来源摘录：个人编程经验与风格

这些内容来自用户提供的印象笔记导出，只作为设计风格来源材料。使用时提炼成通用做法，不要绑定到某个具体项目、类名、存档体系或工具名。

## 基础设计

- 基于类似模块中心的方式做业务逻辑开发。
- 尽可能让一个类只干一种事情。
- 动效、渲染、特效尽可能可以在外部编辑，而不是写死在程序里，方便策划和美术调整，例如 allinone、playtween 一类工具或资源流程。
- 使用 AI 工具时，让 AI 辅助查现有实现、拆设计、确认 API 和生成代码。

## 模块中心

- 模块中心作为功能中心存在。
- 它包含游戏或应用的所有模块以及主要功能入口。
- 所有业务模块通常继承或遵循统一的模块基类/约定。

通用化理解：不同项目可以用模块注册表、服务容器、组合根、依赖注入或功能路由替代具体的 `ModuleControl` 命名。

## 数据与表现

- 数据状态和表现反馈要分离。
- 先设计数据是什么、如何变化、由谁持有最终状态。
- 再设计 UI、动画、特效、音效、场景对象如何响应数据变化。
- 表现层通过动作列表进行：把表现反馈组织成可排序、可组合、可跳过、可回放的动作序列，再由表现层统一调度执行。
- 表现层可以做临时预览和过渡，但不应反向决定正式业务数据。

## 业务代码结构

- 基于事件做低耦合设计。
- 每个模块按需要声明或使用自己的事件中心、消息通道、回调列表或观察机制。
- 使用集中定义的事件名或消息类型保存事件名，保证可搜索和可维护。

## 代码用途类型

- 引擎或基础框架。
- 第三方插件。
- 项目的通用工具。
- 游戏或应用的业务逻辑：
  - 模块。
  - 场景实体组件。
  - UI。

通用化理解：写代码前先判断它属于哪一层，避免业务逻辑污染工具层、UI 持有正式状态、通用能力依赖具体业务。

## 业务逻辑开发流程

1. 阅读策划案或产品需求，先完成定案、讨论和交流。
2. 整理现有功能，判断是否可以复用、整合或扩展。
3. 确定配置方式：
   - 配置表字段以及自动作用。
   - 项目内需要什么样的可视化或编辑器配置。
   - 现有功能是否存在整合方案。
4. 制作配置和工具：
   - 配置表映射。
   - 运行时组件。
   - 行为树节点。
   - 扩展行为命令系统。
   - 配置工具。
   - 编辑器扩展、编辑器工具、编辑器快捷键。
5. 新增对应 UI。
6. 新增或修改对应模块。
7. 制作特效、动效和其他表现反馈。

## 工作方式补充

- 先看实际卡住的地方，再提出几个做法，不要一开始就套固定架构。
- 对复杂架构和重构，先列 1-3 个做法，再和用户问到能直接开工。
- 提问要围绕会改变做法的问题：状态归谁管、表现动作列表怎么排、配置从哪里来、别的模块最少需要知道什么、以后替换 UI/资源/存储会不会牵动业务逻辑。
- 稳定概念要起清楚名字并写下来；被明确拒绝且以后也会影响判断的决定要记录，避免以后重复提出。
